## What's actually missing or undercooked

### 🔴 High Priority (would bite you soon)

**1. No token refresh for LinkedIn OAuth**
The `refresh_token_encrypted` column exists on `User` but is never used. LinkedIn access tokens expire in **60 days**. You have no code that checks `token_expires_at` or calls LinkedIn's refresh endpoint. Currently, when a user's write token expires, they just get a silent failure on carousel publish. You need:
- A pre-flight check in `CarouselService.publish_to_linkedin()` that detects expiry
- Auto-refresh via `refresh_token_encrypted` before attempting the LinkedIn call
- A clean 401 response that tells the frontend "write account needs re-authorization" vs "unknown error"

**2. No schema migration path (Alembic)**
You have `create_all()` on startup. This works pristinely for a new install but has no story for evolving the schema on a live database. The `ON DELETE CASCADE` fix we just made to the model — that will never apply to a running prod DB automatically. As soon as you have real posts/users in production, any schema change requires manual DDL or a data wipe. You need Alembic, even a basic one-time baseline migration.

**3. `datetime.utcnow()` deprecation in models**
Five models (`sales.py`, `llmops.py`, `enterprise.py`) use `datetime.utcnow` which is deprecated in Python 3.12+ and will be removed. This produces 9+ warnings in the test suite. Should be `datetime.now(timezone.utc)`.

---

### 🟡 Medium Priority (matters at scale or reliability)

**4. No `POST /api/v1/auth/logout` endpoint**
Logout is currently 100% client-side (clear localStorage). If a JWT is compromised or a session needs forceful invalidation, there's no server-side mechanism. The `jti` claim is already in the token — a Redis or Postgres blacklist checked in `get_current_user` would complete this.

**5. No ingestion worker in the Docker Compose stack**
The `linkedin-read-flow` service that actually scrapes creator posts isn't in any of the compose files. The Creator Radar feature is present in the API but the feed won't populate unless you run the ingestion service separately. New operators will be confused why the Action Desk is empty.

**6. Pipeline features (Career, Sales, Talent, Enterprise) are UI shells**
These four modules have real endpoints but the underlying data is either seeded mock data or sourced from the ingestion worker. There's no clear "how do I get real data into here" path documented for an operator. These should either be clearly marked as "requires ingestion worker data" or have an admin seed/import endpoint.

**7. No structured error codes**
All errors return `{ "detail": "string" }`. This makes frontend error handling hard — the UI has to parse strings to decide whether to show "re-connect LinkedIn" vs "try again" vs "contact support". A proper error body like `{ "code": "WRITE_TOKEN_EXPIRED", "detail": "..." }` would let the frontend handle errors programmatically.

---

### 🟢 Low Priority (polish / future-proofing)

**8. Test coverage for auth flows is zero**
The 129 tests cover services and endpoints well but there are **no tests for the auth controller** (OAuth initiate, callback, `/me`, write-flow). This is your most security-critical code and it's untested. A mock-LinkedIn integration test for the full login flow would be valuable.

**9. No graceful shutdown / connection draining**
The FastAPI `lifespan` exists but doesn't explicitly close the async DB engine on shutdown. Under Docker `docker stop`, Postgres connections may be left open. Add `await engine.dispose()` in the lifespan cleanup phase.

**10. PDF storage is ephemeral**
Carousel PDFs are written to `/tmp/carousel_pdfs/` inside the container. A container restart wipes them — the carousel asset row still exists with `pdf_url = "file:///tmp/..."` but the file is gone, causing a `FileNotFoundError` on publish. In production, this means a re-generate is required after any deploy. Should mount a persistent volume or use object storage (S3/GCS).

**11. No rate limit enforcement in code**
`config.yaml` defines rate limits (`max_per_day_free: 5`) but there is no middleware or repository code actually enforcing them. The limits are documented but non-functional. A simple `rate_limit_check(user_id, action_type, db)` helper before AI engine calls would close this.
